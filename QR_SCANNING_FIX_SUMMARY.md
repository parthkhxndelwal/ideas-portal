# QR Scanning Fix - Solesta Scanner App

## Problem Identified
The scanner app was rejecting QR codes generated by the Solesta web app with the error: **"the QR isn't valid for solesta!"**

## Root Cause
The issue was in the **AES-256-CBC decryption compatibility** between two different cryptography libraries:
- **Backend (Server)**: Uses Node.js `crypto` module
- **Mobile App**: Uses CryptoJS library

While both use AES-256-CBC encryption, CryptoJS's `AES.decrypt()` method has a specific expectation for how the encrypted data should be formatted.

### The Issue in Detail

**Backend (qr-generator.ts)** encrypts QR data like this:
```typescript
// Node.js crypto module
const cipher = crypto.createCipheriv('aes-256-cbc', key, iv)
let encrypted = cipher.update(data, 'utf8', 'hex')
encrypted += cipher.final('hex')
// Result: "iv_hex:encrypted_hex"
```

**Frontend (crypto.ts - OLD VERSION)** was trying to decrypt like this:
```typescript
// CryptoJS - INCORRECT approach
const decrypted = CryptoJS.AES.decrypt(encrypted, key, {
  iv: iv,
  mode: CryptoJS.mode.CBC,
  padding: CryptoJS.pad.Pkcs7,
})
```

The problem: CryptoJS.AES.decrypt expects the encrypted data as a **WordArray object** with a `ciphertext` property, but was receiving a raw hex string.

## Solution Implemented

Fixed the `decryptAES()` function in `App/crypto.ts` to properly format the encrypted data for CryptoJS:

```typescript
export function decryptAES(encryptedData: string): string {
  try {
    const parts = encryptedData.split(':')
    if (parts.length !== 2) {
      throw new Error('Invalid AES encrypted data format')
    }

    const iv = CryptoJS.enc.Hex.parse(parts[0])
    const encryptedHex = parts[1]
    const key = CryptoJS.enc.Utf8.parse(AES_ENCRYPTION_KEY.padEnd(32, '0').slice(0, 32))

    // KEY FIX: Parse the hex encrypted data as a WordArray
    const encrypted = CryptoJS.enc.Hex.parse(encryptedHex)

    // KEY FIX: Pass encrypted data as { ciphertext: encrypted } object
    const decrypted = CryptoJS.AES.decrypt(
      { ciphertext: encrypted },  // <-- Correct format for CryptoJS
      key,
      {
        iv: iv,
        mode: CryptoJS.mode.CBC,
        padding: CryptoJS.pad.Pkcs7,
      }
    )

    return decrypted.toString(CryptoJS.enc.Utf8)
  } catch (error) {
    console.error('AES decryption error:', error)
    throw new Error('Failed to decrypt AES data')
  }
}
```

### Key Changes:
1. **Line 22**: Convert hex encrypted data to CryptoJS WordArray: `CryptoJS.enc.Hex.parse(encryptedHex)`
2. **Line 25-26**: Pass encrypted data as object with `ciphertext` property instead of raw string

### QR Generation (Backend)
```
Backend generates QR for participant:
1. Create data: "participant_solesta_${referenceId}"
2. Encrypt with AES-256-CBC (Node.js crypto)
3. Format: "iv_hex:encrypted_hex"
4. Encode in PNG QR code
```

### QR Scanning (Mobile App) - NOW FIXED
```
Mobile app scans QR:
1. Read QR code data (encrypted format)
2. Call decryptQRData() in crypto.ts
3. Try AES decryption (now WORKS correctly)
   ├─ Parse IV from hex
   ├─ Parse encrypted data to WordArray
   ├─ Use CryptoJS.AES.decrypt() with correct object format
   └─ Return plaintext: "participant_solesta_${transactionId}"
4. Extract transactionId from plaintext
5. Validate transactionId against cached registrations
6. Record scan if valid
```
